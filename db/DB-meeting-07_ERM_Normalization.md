## 1. E-R 모델 (Entity-Relationship Model)
E-R 데이터 모델은 데이터베이스의 전체적인 논리적 구조(Schema)를 시각적으로 명세하여 데이터베이스 설계를 용이하게 하기 위해 개발되었습니다.

### 1.1. 기본 개념
*   **개체 집합 (Entity Sets)**
    *   **개체 (Entity)**: 다른 객체들과 구별되는 구체적인 대상 (예: 특정 사람, 회사 등)
    *   **개체 집합**: 동일한 성질을 공유하는 같은 타입의 개체들의 모임
    *   **속성 (Attributes)**: 개체 집합의 모든 멤버가 가지고 있는 설명적 특징 (예: 강사 개체의 ID, 이름, 급여 속성)
    *   **기본 키 (Primary Key)**: 개체 집합 내에서 각 멤버를 고유하게 식별할 수 있는 속성들의 부분 집합
*   **관계 집합 (Relationship Sets)**
    *   **관계 (Relationship)**: 여러 개체 간의 연관성
    *   개체 집합들에서 각각 추출한 개체들 간의 수학적 릴레이션으로 표현됨 (예: `advisor` 관계 집합)

### 1.2. 매핑 카디널리티 (Mapping Cardinalities)
한 개체가 관계 집합을 통해 연관될 수 있는 다른 개체의 수를 나타냅니다.
*   **유형**: 1:1 (One-to-one), 1:N (One-to-many), N:1 (Many-to-one), N:M (Many-to-many)
*   **E-R 다이어그램 표현**:
    *   방향이 있는 선 (→): "1" 을 의미
    *   방향이 없는 선 (—): "다수 (Many)" 를 의미

### 1.3. 참여 제약 조건 (Participation Constraints)
*   **전체 참여 (Total Participation)**: 이중선으로 표시하며, 개체 집합의 모든 개체가 적어도 하나의 관계에 참여해야 함 (예: 모든 학생은 지도교수가 있어야 함)
*   **부분 참여 (Partial Participation)**: 일부 개체는 관계에 참여하지 않을 수 있음 (예: 지도 학생이 없는 강사)
*   **복잡한 제약조건 표현**: 선에 `min..max` 형태로 최소 및 최대 카디널리티를 표시 (예: `1..1`, `0..*`)

### 1.4. 기본 키 (Primary Keys)
*   **개체 집합의 기본 키**: 엔티티들을 서로 구별할 수 있게 해주는 속성 집합. 어떤 두 엔티티도 모든 속성에 대해 완전히 같은 값을 가질 수 없음.
*   **관계 집합의 기본 키**: 관계에 참여하는 개체들의 기본 키의 합집합 및 관계 자체가 갖는 속성으로 구성됨.
    *   **N:M 관계**: 참여 개체들 기본 키의 합집합
    *   **1:N / N:1 관계**: "다수(Many)" 측의 기본 키
    *   **1:1 관계**: 참여 개체 중 어느 쪽의 기본 키를 선택해도 무방

### 1.5. 약한 개체 집합 (Weak Entity Sets)
*   독자적으로 존재할 수 없고, 다른 개체(**식별 개체**, Identifying Entity)의 존재에 의존하는 개체 집합.
*   고유한 기본 키가 없으며, 약한 개체들을 구분하기 위한 **구별자(Discriminator)** 속성을 가짐.
*   **약한 개체의 기본 키** = 식별 개체의 기본 키 + 약한 개체의 구별자

## 2. 정규화 (Normalization)

### 2.1. 정규화의 필요성과 분해
*   **데이터 섞임의 문제점 (Anomaly)**: 서로 다른 데이터를 무분별하게 하나의 릴레이션으로 합치면, 데이터 중복으로 인해 저장 공간이 낭비되고 일관성 문제가 발생함.
    *   **삽입 이상 (Insertion anomaly)**: 새 레코드를 삽입할 때 원치 않는 중복 데이터를 억지로 넣어야 함.
    *   **삭제 이상 (Deletion anomaly)**: 특정 데이터를 삭제할 때 연관된 유용한 다른 데이터까지 함께 소실됨.
    *   **갱신 이상 (Update anomaly)**: 정보 변경 시 모든 중복 레코드를 하나하나 찾아서 수정해야 함.
*   **해결책 - 분해 (Decomposition)**:
    *   릴레이션 스키마를 쪼개어 중복을 제거하는 과정이 바로 정규화임.
    *   **무손실 분해 (Lossless Decomposition)**: 분해된 릴레이션들을 조인($\bowtie$)했을 때 정보의 손실 없이 원래의 릴레이션으로 완벽히 복원되는 분해. (정규화는 무손실 분해를 보장해야 함)
    *   *손실 분해 (Lossy Decomposition)*: 조인 결과가 원래 릴레이션의 초과 집합(Superset)이 되어 원본을 알아낼 수 없는 잘못된 분해.

### 2.2. 함수적 종속성 (Functional Dependencies)
*   임의의 두 튜플이 속성 $A$ 에서 일치하면 속성 $B$ 에서도 반드시 일치한다는 제약 조건 ($A \rightarrow B$).
*   기본 키(Primary Key) 개념의 일반화임.

### 2.3. 정규형의 종류 (Normal Forms)
정규화는 순차적인 과정이며, 상위 정규형이 되려면 하위 정규형 조건을 만족해야 합니다.

1.  **제1정규형 (1NF: First Normal Form)**
    *   릴레이션의 모든 속성 값은 분할할 수 없는 **원자 값(Atomic Value)**이어야 함 (예: 리스트, 구조체 사용 금지).
    *   각 컬럼은 동일한 데이터 도메인을 가져야 하며, 고유한 이름을 가져야 하고 중복된 행이 없어야 함.

2.  **제2정규형 (2NF: Second Normal Form)**
    *   1NF를 만족해야 함.
    *   **부분 종속성(Partial Dependency) 제거**: 기본 키가 복합 키일 때, 기본 키의 일부 속성에만 종속되는 일반 속성이 없어야 함. (모든 일반 속성은 기본 키 전체에 완전 종속되어야 함)

3.  **제3정규형 (3NF: Third Normal Form)**
    *   2NF를 만족해야 함.
    *   **이행적 종속성(Transitive Dependency) 제거**: 기본 키가 아닌 일반 속성이 다른 일반 속성을 결정(종속)하는 일이 없어야 함. ($A \rightarrow B$, $B \rightarrow C$ 제거)

4.  **BCNF (Boyce-Codd Normal Form / 3.5NF)**
    *   3NF를 만족해야 함.
    *   모든 함수적 종속성 $A \rightarrow B$ 에 대해, **결정자인 $A$는 반드시 슈퍼 키(Super Key)**이어야 함.
    *   기본 키가 아닌 속성이 기본 키의 일부를 결정하는 경우를 방지함.

5.  **제4정규형 (4NF: Fourth Normal Form)**
    *   BCNF를 만족해야 함.
    *   잘못된 스키마 설계로 인해 3개 이상의 속성 간에 발생하는 **다중값 종속성(Multi-valued Dependency)을 제거**함.

### 2.4. 기타 설계 고려사항
*   **역정규화 (Denormalization for Performance)**
    *   빠른 검색 성능(조인 연산 최소화)을 위해 의도적으로 정규화를 포기하고 속성을 합치는 것.
    *   업데이트 시간 증가, 오류 발생 가능성, 여분의 저장 공간 필요 등의 단점이 있음. (대안으로 Materialized View 활용 가능)
*   **정규화로 해결되지 않는 이슈**
    *   모든 설계 문제가 정규화로 잡아지지는 않음. 예를 들어, 크로스탭(Crosstab) 형태(연도별로 컬럼이 늘어나는 구조)는 BCNF를 만족하더라도 좋은 설계가 아님. 이를 올바른 스키마 구조(예: `company_id`, `year`, `amount`)로 재설계해야 함.
